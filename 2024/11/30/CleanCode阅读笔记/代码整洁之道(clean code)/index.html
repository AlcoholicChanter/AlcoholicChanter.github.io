Clean Code Notes

要点总结

《代码整洁之道》这本书重点强调了代码的优美性(易阅读、易理解、易维护、易拓展，四易)。指出了优美代码在整个软件设计中的重要地位，以及该从哪些方面考虑书写代码的优美。

其中包含了一些原则(some principles of clean code include)：

1、有意义的命名(Meaningful Name)

2、尽可能短小的函数(Small Functions)

3、避免重复(Avoid Duplication)

4、格式化的代码(Code formatting)

5、定期重构代码(Refactoring)

该书并不是一本某项具体技术的实用书籍，它是一本思想教育书籍！意在指导读者改变设计思想，从而使得设计出来的代码更加优美。

1、有意义的命名(Meaningful Names)

1.1、使用有意义的命名，不过需要保证命名的简洁性(Be Descriptive,But Concise)。有意义的命名，首先就需要保证命名的内容要名副其实。一定要避免‘挂羊头卖狗肉’的行为！因为这样不仅仅会让其他代码阅读者很难理解，过了一段时间之后，也许作者自己也很难琢磨透这个命名的含义。

1.2、在一个工程中，尽量统一所有coder的命名规范(Use Consistent Naming Conventions)。例如统一使用 大驼峰命名法 或者 小驼峰命名法 或者 蛇形命名法 等等。但不管使用哪种命名法，都请务必要做到保持一致！不然在阅读代码工程中将会无形的增加许多确定的各种命名含义的成本。

本人现在习惯使用的方式是：

类名、函数名 => 大驼峰命名法，

成员变量=>小驼峰命名法，

静态函数=>大驼峰命名法+_连接单词

静态变量=>小驼峰命名法+_连接单词

按照上述这样，可以简单的从命名就可以完成对于各种不同成员的区分。

虽然在现代IDE的强大辅助性下，这些内容显得没那么重要。不过，既然打算优雅的编写代码，那么细节仍然处于重要地位。

1.3、避免使用通用名称(Avoid Generic Names)。避免使用类似于data、temp、tmp、things等代词，尽量使用具体要处理的数据代词。如果是通用类型的成员，最好能够抽象出通用的描述词语，如果不能具体的描述，用注释简单标注一下函数的用途。

1.4、使用有意义的缩写(Use Meaningful Abbreviationsviations)。在团队中统一并规范使用代词缩写，可以考虑专门维护一份文件，用于存储代词缩写所代表的含义，这样减少了整体的命名长度。

1.5、优先考虑能否使用计算机术语、数学术语、设计模式术语等去进行命名，如果有的话。否则，使用具体领域下的术语(Use Domain-Specific Terms)。在这里需要强调的是，**只有同行者才会去阅读代码！**所以，优先考虑用有交集的领域术语命名，这样理解起来会直白很多。如果不能用这些领域的术语，那么也可以考虑用具体业务的术语，来进行命名。这样至少在询问相关的业务人员之后，也能明白大概的代码意思。

1.6、和同行者一起讨论命名(ask for feedback)！在无法确定的时候，可以尝试和同行者讨论某种情况下的命名。这是最简单有效的方式，前提是消耗你的☕️。

1.7、多加思考，以及对变量进行重命名(Practice And Rename)。在短时间内，其实很难设想出一个足够符合语意的名词或者短语去进行描述。可以先选择一个能足够表达含义的词进行命名，在后续的时间里再去考虑这个命名(为什么需要考虑呢？因为描述不够清晰！)，总会有一个足够让你满意的命名在你持续思考之后反馈给你。

2、精简的函数(Small Function)

2.1、单一职责原则(Single Responsibility Principle 简称 SRP)。对于SRP，最简单的理解其实就是一个函数只做一件事情。比如：接收数据并存储是一件事情、格式化一段字符是一件事情、将字符渲染到文本上也是一件事情。

​ 对于上述举例，进行反向思考。如果，将接收数据并存储、格式化一段字符、将字符渲染到文本上，这三件事情都放在一个函数里处理，那么这个函数就已经有了god function的雏形了！现在假设需要改动传入数据、需要改动格式化字符的样式、需要改动渲染的文本对象，都需要改动这个函数！…..变更意味着风险！对于维护程序的稳定性来说，需要最大程度的降低变更代码的风险。

2.2、避免较长的switch和if….else if…结构。

Approach (方法) When to Use (使用场景) Benefits (优点)
Lookup Map/Object (查找表) Mapping values to results or actions (值到结果或操作的映射). Simplifies branching logic (简化分支逻辑).
Strategy Pattern (策略模式) When actions depend on conditions (根据条件执行不同操作). Modular and extensible (模块化且易扩展).
Polymorphism (多态) Complex logic that varies by type (复杂逻辑按类型变化). Clear and reusable design (清晰且可复用的设计).
Functional Composition Modular and reusable logic (模块化且可复用的逻辑). Avoids deep nesting (避免深度嵌套).

2.3、将异常处理当成一件事情！

2.4、尽量避免重复！一个类似的action存在一个就够了，如果没办法完成抽象。。。。。那么请在一段时间之后将它重构一遍。

2.5、避免结构化编程。结构化编程的思想，适合处理大的函数，大的类。如果设计足够面向对象，理论上是不会存在大的函数，大的类。

3、有效的注释(Effective Comments)

“Don’t comment bad code—rewrite it.”

别给糟糕的代码添加注释了，重构它吧。

代码本身也有阅读的意义。即使是面向阴间的硬编码，再通过令人绝望的解析之后，也是可以进行阅读的。但这和阅读优美的代码是两码事，给人的感受也是完全不同的。前者就好像去了一个看起来能吃的餐馆，点了一份看起来看行的菜，最后吃起来味同嚼蜡；后者则是装在一个精致盒子里的美味蛋糕，首先得费点心思打开精致的盒子，然后才能吃到美味的蛋糕！

目前看来，注释存在的意义最好只有以下情况：

  • 法律信息
  • 某类，某函数用法示例
  • 行为的意图解释
  • 危险行为的警告或者标识
  • TODO标识
  • API标注

并不一定需要注释存在的情况：

  • 和类名、函数名表达意思一样的注释
  • 表述目的不明确的注释
  • 日志式的注释
  • 当作特殊标识的注释

一定不需要存在注释的情况：

  • 远古注释-但注释下的内容经常改动(容易让阅读者产生误解的注释)
  • 被注释的代码-有版本控制器，为什么还要注释掉呢…🤔
  • 大量内容的注释-比如每一次修改的注释，其实读者需要的仅仅只是当前所进行行为的注释就够了

能够帮助减少上面 并不一定需要存在 和 一定不需要存在注释 的方式其实很简单，做好 有意义的命名 以及 精简的函数！

4、保持一种代码格式(Keeping A Coding Format)

尽量的保持一种编码的格式，从上至下也好，相似的功能放在一起也好。这样可以让经常阅读你代码的人有一种熟悉感。这种感觉并非能够产生实际上的意义，不过也许会让读者减轻一些面对陌生代码时焦急的心理负担。

5、如何处理错误(How to deal Error)

5.1、任何时候，请把处理错误当成一件事情！它应当有一个专门的封装函数，来区分需要捕获异常的代码，以及捕获到异常之后的处理代码。

5.2、当捕获到异常之后，尽可能的将导致异常的相关信息输出，这将成为后续分析的总要信息依据。

5.3、当异常发生的时候，需要尽可能的避免异常导致整个程序被阻塞，使用简单的方法，先保证程序不会挂掉。

6、如何与第三方系统保持边界(How to Keep Boundaries with Third-Party Systems)

6.1、对第三方进行接口抽离，保证该第三方系统的接口都在一个可控的范围内。

6.2、限制第三方接口的访问权限，杜绝让第三方接口充斥核心业务层

6.3、提前设计好测试的mock数据，在第三方系统进行更新之后，逐一进行测试

7、如何进行测试((How to Perform Testing))

7.1、引入单元测试，针对各个系统的最小功能点进行单元测试，保证各个系统最小计算正确。

7.2、引入集成测试，针对各个系统独立整体、各个系统之间的数据交互进行集成测试，保持系统整体性正确。

7.3、引入自动测试流，通过CI/CD自动化模式，加之程序员对自己mock数据内容，进行自动化验证数据。

key point：推进整个项目使用TDD测试模式。

what is TDD? TDD stands for Tesk-Drive Development.

通过使用先设计测试数据，如何再编写代码，接着再保证编写的代码能够让数据通过测试，最后再去优化过程代码。

虽然这种模式会让开发周期在一定程度上延长，但其带来的好处是：

1、保证能够枚举出来的异常情况的处理，

2、保证代码的正确性，

3、保证代码后期维护、优化时的正确性。

这三点，作为一个开发来说，简直是梦寐以求的开发环境。

8、精简的类(Lean Class)

8.1、杜绝god class的出现，一个class不应该尝试做所有事情。每个class应该拥有单一的职责，并且将这件事情做到极致。

8.2、贯彻单一职责设计思维(SRP)，这样能够增强项目的可拓展性和易维护性。

9、如何设计一个系统(How to Design a System)

9.1、遵循五大面向对象设计原则(SOLID)：单一职责原则(S,SRP,Single Responsibility Principle)、开闭原则(O,OCP,Open/Closed Principle)、里氏替换原则(L,LSP,Lisko Substitution Principle)、接口隔离原则(I,ISP,Interface Segregation Principle)、依赖倒置原则(D,DIP,Dependency Inversion Principle)。

9.2、使用模块化设计，使用插拔式去对模块进行加载移除的管理。

9.3、模块之前的交互，需要抽离专门的接口，避免模块与模块之前的耦合。

10、如何控制迭代(How to Control Iterations)

10.1、贯彻敏捷开发的思想，先区分需求等级，再根据等级去划分大版本、小版本。

10.2、规划定期复盘，对前段时间进行复盘总结，并且在下一阶段要努力去落实复盘成果。

10.3、尽可能收集用户反馈，根据用户的反馈来定制版本的更新迭代。

11、并发/异步编程(Concurrency Programming)

11.1、避免可共享的数据状态，使用类似于加锁、Await等手段，保证数据不会被多个并发同时更改。(并发和异步都需要注意⚠️)

11.2、使用异步来设置程序的阻塞。例如await Promise

11.3、将并发/异步集成为一个系统，封装底层内容，统一管理。
